Goal:
Make lead deletion succeed safely, or return a clear handled validation error if deletion is intentionally blocked by related records. Do not allow an unhandled 500.

STRICT SCOPE:
Only investigate and fix lead deletion behavior and related dependency cleanup / constraint handling.

Do not modify:
- follow-up scheduling logic
- task completion modal
- task validation rules
- timezone logic
- task history/activity display
- sales pipeline stage logic
- lead qualification / not interested logic
- unrelated layouts
- unrelated forms
- unrelated code review findings

Only edit the minimum files necessary.
After implementation, stop.

Required investigation:
1. Audit the backend delete flow for leads used by both Sales Pipeline and CRM.
2. Identify exactly which related records are causing the 500. Check at minimum:
- follow-up tasks (open and completed)
- activity timeline entries
- history entries / lead notes
- linked opportunities
- linked contacts
- linked companies
- source lead / source mapping records
- any foreign key constraints triggered during delete

Required behavior:
3. If the product expects hard delete for leads, make deletion handle related records safely so the lead can be deleted without server error.
Use the cleanest existing pattern already used elsewhere in the app:
- cascade delete related records where appropriate, or
- detach / null references where appropriate, or
- delete child records first in the correct order

4. If there is a valid reason deletion must be blocked, return a clear handled error response instead of 500.
Example:
“This lead cannot be deleted until related records are removed.”

6. Reuse existing delete / cleanup patterns if similar entities already support safe deletion.
7. Keep the fix tightly scoped to lead deletion and dependent-record cleanup only.
8. Do not introduce unrelated refactors.
