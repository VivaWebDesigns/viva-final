We need to diagnose the live /my-leads worker isolation bug.

Important observed behavior:
A brand-new extension_worker account signed into a freshly unpacked Chrome extension, but the worker queue showed the same Active/Converted counts and records as another worker.

Extension repo diagnosis found:
- fetchWorkerQueue() calls /api/marketplace/pending-outreach/my-leads
- It uses getAuthHeader()
- When authState === "worker_signed_in", getAuthHeader() should send the worker bearer token
- However, extension fetch calls also use credentials: "include"
- That means vivawebdesigns.com cookies may be sent along with the worker bearer token
- If backend auth resolves cookie/session identity before bearer token identity, an existing admin cookie in Chrome could cause req.authUser to be admin instead of the signed-in extension worker

Expected behavior:
GET /api/marketplace/pending-outreach/my-leads?statusGroup=open
GET /api/marketplace/pending-outreach/my-leads?statusGroup=converted

should return only records where:

createdBy = req.authUser.id

for the signed-in extension_worker.
